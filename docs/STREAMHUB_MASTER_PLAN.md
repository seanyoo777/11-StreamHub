# StreamHub 마스터 기획서 (2026)

**문서 버전:** 1.0  
**단계:** 기획 1단계 (방송 플랫폼 마스터)  
**범위:** 설계·운영·로드맵 초안 (본 문서는 실제 결제·토큰 발행·송출 서버 구축·외부 API 연동을 포함하지 않음)

---

## 1. 프로젝트 개요

### 1.1 비전

**StreamHub**는 저수수료를 전제로 한 **2026년형 실시간 방송 허브**다. 모바일과 PC에서 동일한 품질의 시청·채팅·후원 경험을 제공하고, BJ(스트리머)는 **OBS/XSplit 등 표준 RTMP 송출 도구**로 방송할 수 있는 구조를 목표로 한다.

### 1.2 핵심 가치

| 가치 | 설명 |
|------|------|
| 낮은 수수료 | 플랫폼이 가져가는 정산 비율을 낮추고, BJ·시청자 모두에게 유리한 구조를 설계한다. |
| 도구 호환 | 업계 표준 송출 스택(RTMP 등)을 전제로 하여 전환 비용을 줄인다. |
| 실시간 상호작용 | 채팅·알림·랭킹으로 방송방 내 참여도를 높인다. |
| 확장성 | 게임·모의투자·커뮤니티·AI(OneAI)와의 **이벤트·데이터 계약**만으로 연동 가능한 경계를 둔다. |
| Web3 옵션 | **중앙 스트리밍·계정·포인트**를 기본으로 하되, 향후 **지갑 연동·온체인 증명(설계만)** 등 Web3 요소를 선택적으로 결합할 수 있는 자리를 남긴다. |

### 1.3 본 단계에서의 명시적 제외 (절대 금지 준수)

- 실제 결제(PG, 카드, 앱스토어 결제 등) 연결  
- 토큰 실발행 및 실제 스마트컨트랙트 배포  
- **자체 실시간 송출(인제스트) 서버** 구축·운영 (기획서에서는 **외부 스트리밍/CDN 역할을 “블랙박스”**로 두고 인터페이스만 정의)  
- 외부 API 연동 구현  
- 대규모 코드 작성 (앱은 단계적·최소 구현)

### 1.4 이해관계자

- **BJ/스트리머:** 방송 생성·수익·팬 관리  
- **시청자:** 시청·채팅·내부 포인트 후원  
- **운영/관리자:** 콘텐츠·유저·정산 정책·분쟁 대응  
- **연동 파트너(향후):** 게임·모의투자·커뮤니티·AI 서비스

## 2. 핵심 방송 구조

### 2.1 논리 아키텍처 (고수준)

```
[BJ PC] OBS/XSplit ──RTMP──▶ [송출·변환·배포] ◀── 블랙박스(외부/향후 계약)
                                    │
                                    ├── CDN/재생 URL (HLS/DASH 등)
                                    │
[시청자] 모바일/PC 웹·앱 ──재생──▶ 플레이어
                │
                └── WebSocket/SSE ──▶ [채팅·알림·후원 이벤트 서버]
```

- **송출 경로:** BJ는 RTMP URL + 스트림 키(또는 동등한 자격)만 알면 송출 가능하도록 **인터페이스를 표준화**한다.  
- **재생 경로:** 시청자 앱은 **재생 가능한 매니페스트 URL**과 **방 상태(ON AIR / OFF)** 만 신뢰하면 된다.  
- **상호작용 경로:** 영상과 분리된 **실시간 메시징 레이어**로 채팅·후원·알림을 처리하여 확장과 장애 격리를 용이하게 한다.

### 2.2 방송 “세션” 개념

- **라이브 세션:** 시작 시각·종료 시각·BJ ID·카테고리·연령/등급 메타데이터  
- **스트림 식별자:** 세션당 고유 스트림 ID (재생 URL 생성의 기준)  
- **백업/재연결:** BJ 단말 끊김 시 세션 유지 정책(타임아웃 후 자동 종료 등)을 운영 규칙으로 정의

### 2.3 품질·지연

- **저지연 모드**와 **안정 우선 모드**를 정책으로 구분할 수 있게 설계 (초기에는 한 가지 모드만 지원해도 됨)  
- 모바일은 네트워크 변동에 대비한 **적응형 비트레이트(ABR)** 재생을 전제

## 3. BJ/스트리머 구조

### 3.1 계정·권한

- **BJ 등급:** 일반 시청자 계정과 분리된 **BJ 신청·승인** 플로우(또는 역할 플래그)  
- **방송방 소유:** BJ 1인 N개 방(카테고리별) 가능 여부를 정책으로 결정  
- **모더레이션 권한:** 채팅 금지·타임아웃·강퇴 등 방 단위 권한

### 3.2 송출 설정 UX

- **RTMP 서버 주소·스트림 키** 표시 (복사 버튼, 재발급, 유출 시 폐기)  
- **해상도·비트레이트 가이드** (권장 프리셋: 720p/1080p, 업로드 대역폭 안내)  
- **방송 시작/종료:** 플랫폼에서 “ON AIR” 전환은 **스트림 수신 확인 이벤트** 또는 **BJ 수동 버튼** 중 하나로 통일

### 3.3 수익·정산(기획만)

- **내부 포인트·코인**으로 후원이 집계되고, BJ는 **플랫폼 내부 잔액/정산 대시보드**에서 확인  
- 실제 현금 정산·세무는 **별도 단계·별도 시스템**으로 분리 (본 기획 단계에서는 연결하지 않음)

## 4. 시청자 구조

### 4.1 진입·발견

- **홈:** 추천 라이브, 카테고리, 팔로우 중 라이브  
- **검색:** BJ명·방 제목·태그 (스팸 방지 규칙 필요)  
- **딥링크:** 특정 방 URL로 모바일/PC 공유

### 4.2 시청 경험

- **플레이어:** 전체화면, 화질 선택(가능 시), 자막/자막 확장은 후순위  
- **동시 시청자 수:** BJ에게는 대시보드, 시청자에게는 선택적 표시(과장 방지 정책 검토)  
- **로그인 범위:** 익명 시청 vs 로그인 시 채팅·후원 제한 등 단계적 정책

### 4.3 팬십·구독(선택)

- **팔로우/알림 구독**은 내부 구현으로 가능  
- **유료 구독**은 실결제 없이 **“설계 자리”**만 확보하거나 MVP에서 제외

---

## 5. 후원/포인트 구조

### 5.1 원칙

- 모든 가치 이동은 **플랫폼 내부 포인트/코인**으로만 표현  
- **충전·환전·실제 결제 연동 없음** — 초기에는 **운영 지급·이벤트 보상·테스트 크레딧**으로 순환 구조를 가정

### 5.2 데이터 모델(개념)

- **지감:** 사용자별 포인트 잔액, 코인 잔액(필요 시 이원화)  
- **후원 트랜잭션:** 보낸이·받는이(BJ)·금액·수수료율·메시지(선택)·타임스탬프  
- **수수료:** 플랫폼 수수료 vs BJ 정산액을 **명시적 필드**로 분리해 투명성 확보

### 5.3 저수수료 정책(기획)

- 기본 후원 수수료율을 **낮은 밴드**로 고정하고, 이벤트 기간 한시 인하  
- **누적 후원 랭킹**과 연계한 비금전 보상(배지 등)으로 BJ 동기 부여

### 5.4 Web3 결합(설계 레벨만)

- **온체인 자산 이동 없음**을 전제로, 향후 **지갑 주소 연동(읽기 전용)** 또는 **NFT 보유 증명(오프체인 검증)** 슬롯만 정의  
- 실토큰·실스왑·실브릿지는 본 로드맵 범위 밖

## 6. 채팅/알림/랭킹

### 6.1 실시간 채팅

- **방 단위 채널**, 메시지 타입: 일반·시스템·후원 하이라이트  
- **속도 제한·중복 필터·금칙어(단계적)**  
- **모더레이션 큐:** 신고 누적 시 자동 저속 모드 등

### 6.2 알림

- **인앱:** 팔로우 BJ 라이브 시작, 멘션, 후원 수신(BJ)  
- **푸시:** 모바일 OS 푸시는 별도 인증·인프라 — 초기에는 인앱만

### 6.3 랭킹

- **일간/주간 후원**, **시청자 기여**, **BJ 인기** 등 지표를 **분리된 랭킹 테이블**로 설계해 조작 완화  
- **Anti-abuse:** 동일 디바이스 다계정, 포인트 세탁 패턴 탐지(운영 도구와 함께)

## 7. 모바일/PC UX 방향

### 7.1 공통 정보 구조

- **상단:** 방 제목·BJ 프로필·라이브 배지  
- **중앙:** 플레이어  
- **하단/측면:** 채팅·후원 패널 (PC는 좌/우 분할, 모바일은 탭·바텀시트)

### 7.2 모바일

- **세로 우선:** 한 손 조작, 채팅 입력 시 키보드 대응  
- **절전·백그라운드:** 오디오만 재생 옵션(정책·OS 제약 반영)

### 7.3 PC

- **넓은 채팅열**, **멀티 모니터 BJ용 대시보드** 별도 화면  
- **키보드 단축키**(음소거, 전체화면)는 후순위

### 7.4 접근성·안전

- **자막·대비** 가이드라인 준수 여지  
- **연령 표시·콘텐츠 경고** 배너

## 8. 게임 플랫폼 연동

### 8.1 연동 패턴 (API 없이도 설계 가능한 수준)

- **이벤트 버스 개념:** 게임 서버 → StreamHub **내부 웹훅/메시지 큐(향후)** 로 “킬 달성”, “랭크 변동” 등 이벤트 전달 계약  
- **오버레이:** BJ 방송 화면에 게임 상태를 **OBS 브라우저 소스**로 붙이는 운영 가이드(플랫폼은 URL·토큰만 제공하는 형태도 가능)

### 8.2 MVP 범위

- 방송 메타데이터에 **“게임 태그”** 및 **외부 오버레이 링크(선택)** 필드만 두고, 게임 로직은 게임 쪽 책임으로 한정

---

## 9. 모의투자/OneAI 연동 가능성

### 9.1 모의투자

- **시그널 표시:** 모의투자 포트폴리오 변동 → 방송 중 **시청자용 요약 카드**(지연 허용)  
- **책임 분리:** 투자 조언이 아닌 **교육·엔터테인먼트** 성격 명시, 리스크 고지 문구 템플릿

### 9.2 OneAI

- **채팅 요약·타임라인 하이라이트·BJ 코파일럿(스크립트 초안)** 등 **비실시간·저빈도** 작업부터  
- **개인정보·저작권:** 채팅 로그 학습 여부를 **옵트인**으로 설계

## 11. 서버/CDN 비용 구조 초안

> 송출·트랜스코딩·CDN은 **자체 구축 금지** 전제이므로, 아래는 **외부 과금 모델을 가정한 원가 항목 분해**다. 실제 단가는 계약 시점에 재산정.

### 11.1 비용 항목

| 항목 | 설명 | 과금 드라이버 |
|------|------|----------------|
| 인제스트·라이브 트랜스코딩 | RTMP 수신·인코딩 파이프 | 동시 송출 수, 출력 해상도 수 |
| CDN 출带宽 | 시청자 재생 트래픽 | 동시 시청자, 평균 비트레이트, 지역 분포 |
| 스토리지 | VOD 클립(선택) | 저장 분량, 보존 기간 |
| 실시간 백엔드 | 채팅·WS, 메시지 브로커 | 초당 메시지, 접속 수 |
| DB·캐시 | 유저·방·후원 원장 | 쓰기 TPS, 보관 기간 |
| 관측·로그 | APM, 로그 검색 | 로그량 |

### 11.2 비용 통제 전략

- **동시 시청 상한·ABR 상한**으로 CDN 폭주 완화  
- **클립/VOD**는 초기 비활성 또는 짧은 보존  
- **채팅 샤딩** 및 **메시지 샘플링(비핵심 이벤트)** 으로 WS 비용 절감

### 11.3 수익·수수료와의 관계

- **저수수료**는 BJ 친화적이나 **CDN·인코딩 변동비**와 충돌할 수 있음 → **수수료 + 트래픽 쿼터** 또는 **BJ 등급별 무료 트래픽** 등 **혼합 모델**을 기획안으로 검토

## 12. 개발 단계별 로드맵

| 단계 | 목표 | 주요 산출물 |
|------|------|-------------|
| **0. 기획 고정** | 용어·역할·금지 범위 합의 | 본 문서, 화면 IA 와이어 |
| **1. MVP 방송 소비** | 재생 URL·방 상태·BJ 프로필 최소 | 플레이어 페이지, 방 메타 API(내부) |
| **2. MVP 상호작용** | 채팅·내부 포인트 후원 | WS 채널, 후원 원장, 수수료 필드 |
| **3. BJ 도구** | RTMP 안내·키 발급·대시보드 | 송출 설정 UI, 세션 로그 |
| **4. 운영·안전** | 신고·제재·랭킹 안티 어뷰즈 | 어드민 최소 기능 |
| **5. 연동 확장** | 게임/모의투자/OneAI **계약 스펙** | 이벤트 스키마 문서, 목업 |
| **6. Web3 옵션** | 지갑 연동·증명 UI **목업 수준** | 설계 RFC (실발행 없음) |

각 단계는 **실결제·실발행·외부 API·대규모 코드** 없이 검증 가능한 범위로 쪼갠다.
